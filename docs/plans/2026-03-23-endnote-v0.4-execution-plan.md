# EndNote v0.4 执行计划

> 目标：在不破坏现有 Zotero 主线的前提下，为 `thesis-skills` 增加 EndNote import-first 能力，并提供稳定的导入、映射、Bib 输出、报告与测试基线。

关联文档：

- 产品/架构说明：`docs/plans/2026-03-23-endnote-v0.4-product-architecture.md`
- 原始汇总版草案：`thesis-skills-endnote-v0.4-plan.md`

---

## 1. 执行原则

1. **主线安全优先**：先保护 Zotero workflow，再扩 EndNote。
2. **phase-by-phase**：不允许一次性做大爆炸式改造。
3. **样例先行**：先准备 XML / RIS 样例与回归基线，再做核心实现。
4. **兼容优先于抽象完整**：`citation_mapping.py` 若要扩展，必须先保证旧 API 与旧格式继续可用。
5. **import-first only**：严格限制在 v0.4 范围内，不做 EndNote Word parser。

### 1.1 开工前保护要求

在真正进入实现前，必须先建立可回滚点与隔离工作机制。

## Phase 1：最小通用模型与解析层

### 目标

先打通 `EndNote 文件 -> 内部统一对象`，此阶段不处理 mapping 重构。

### 必做文件

- `core/citation_models.py`
- `core/endnote_xml.py`
- `core/endnote_ris.py`
- `core/canonicalize.py`

### 模块契约

#### `core/citation_models.py`

至少定义：

```python
@dataclass
class PersonName:
    family: str = ""
    given: str = ""

@dataclass
class CanonicalRef:
    source_system: str
    source_id: str | None
    canonical_id: str
    entry_type: str
    title: str | None
    authors: list[PersonName]
    year: str | None
    doi: str | None
    isbn: str | None
    issn: str | None
    journal: str | None
    booktitle: str | None
    publisher: str | None
    langid: str | None
    raw: dict[str, Any]

@dataclass
class ImportResult:
    source_file: Path
    source_format: str
    refs: list[CanonicalRef]
    warnings: list[str]
    errors: list[str]
```

#### `core/endnote_xml.py`

- 使用标准库 `xml.etree.ElementTree`
- 负责从 XML 提取字段
- 返回 `ImportResult`
- 未知类型与缺字段写 warning，不抛 silent error

#### `core/endnote_ris.py`

- 支持 RIS 常见 tags：`TY TI T1 AU A1 PY Y1 JO JF T2 PB DO SN SP EP VL IS UR AB LA ER`
- 支持 continuation line 与多作者
- 返回 `ImportResult`

#### `core/canonicalize.py`

- DOI 规范化
- 标题标准化
- 作者名标准化
- `langid` 推断
- `canonical_id` 生成

### 退出标准

- XML 与 RIS 样例都能解析
- 中英文标题与多作者可保留
- 缺 DOI / 未知 type 不崩溃
- 已有 parser 与 canonicalize 的基础测试

### 明确任务包

#### Task 1.1：定义最小通用数据模型

**目标**：先确定 `CanonicalRef`、`PersonName`、`ImportResult` 的字段边界。  
**关键要求**：不引入与 Zotero 强绑定的字段命名。

#### Task 1.2：实现 XML parser 的最小闭环

**目标**：先让 XML 样例可稳定转为 `ImportResult`。  
**重点**：title、authors、year、journal、publisher、doi、isbn/issn、raw 保留。

#### Task 1.3：实现 RIS parser 的最小闭环

**目标**：让 RIS 样例在多作者、continuation line、常见 tags 下可稳定解析。  
**重点**：TY/ER 边界与 continuation line。

#### Task 1.4：实现 canonicalize 最小能力

**目标**：只先做导入链路真正需要的标准化。  
**包含**：DOI、标题、作者、langid、canonical_id。  
**不包含**：过度复杂的跨来源统一语义设计。

#### Task 1.5：补基础单元测试

**目标**：在进入 Bib 输出和 CLI 前，先锁定 parser / canonicalize 行为。  
**建议测试文件**：

- `tests/test_endnote_xml.py`
- `tests/test_endnote_ris.py`
- `tests/test_canonicalize.py`

## Phase 2：渲染、导入 CLI 与报告

### 目标

在不大动主线的前提下，先形成可运行的 EndNote import-first CLI。

### 必做文件

- `core/bib_render.py`
- `core/match_refs.py`
- `00-bib-endnote/import_library.py`
- `00-bib-endnote/check_endnote_export.py`

### 要求

#### `core/bib_render.py`

实现：

- `canonical_ref_to_bibtex(ref: CanonicalRef, latex_key: str) -> str`
- `render_bib_file(refs: list[CanonicalRef], mapping: dict[str, str]) -> str`

要求：

- 输出稳定、可读、可 diff
- 尽量补 `langid`
- DOI 清洗后写出
- entry type 归一化到有限集合

#### `core/match_refs.py`

实现：

- 去重
- 匹配优先级：DOI > ISBN/ISSN+title > title+year+first author
- 低置信度只 warning，不自动合并

#### `00-bib-endnote/import_library.py`

建议 CLI：

```bash
python 00-bib-endnote/import_library.py \
  --project-root <latex-project> \
  --input <file> \
  --format auto \
  --apply \
  --report <path>
```

主流程：

1. 自动识别输入格式
2. 解析
3. 规范化
4. 去重
5. 复用或分配 `refNNN`
6. 写入 `ref/refs-import.bib`
7. 更新 `ref/citation-mapping.json`
8. 输出 report

其中 BibTeX 输入的落地要求为：

- 至少可作为兼容输入被接受
- 即便字段不完美，也不能导致整个流程失败
- 字段缺失、类型异常、格式污染优先进入 warning / report
- 除非复杂度超出预期，否则不新增专门的 BibTeX parser 模块

#### `00-bib-endnote/check_endnote_export.py`

检查：

- XML / RIS 格式合法性
- 缺 title / year / author
- DOI 空格污染
- 未知 entry type
- 明显重复项

### 报告至少包含

- total refs
- imported refs
- deduped refs
- reused mapping count
- new mapping count
- warnings
- unsupported types
- missing fields

### 退出标准

- 可以 dry-run
- 可以 apply
- 可生成 `refs-import.bib`
- 可输出 JSON report

### 明确任务包

#### Task 2.1：实现 bib_render 输出稳定性

**目标**：生成可读、可 diff、字段顺序稳定的 BibTeX。  
**重点**：`
refNNN` 作为 key、langid 补充、DOI 清洗、entry type 限定。

#### Task 2.2：实现 match_refs 的高置信规则

**目标**：只实现 v0.4 真正需要的去重能力。  
**重点**：DOI exact 自动合并；低置信仅 warning。

#### Task 2.3：实现 import_library.py 的 dry-run 模式

**目标**：先在不写文件的情况下跑通全链路。  
**重点**：格式自动识别、解析、规范化、去重、拟定 mapping、生成 report。

#### Task 2.4：实现 import_library.py 的 apply 模式

**目标**：确认 dry-run 稳定后，再允许写入 `refs-import.bib` 与 mapping。  
**重点**：重复运行不重复追加。

#### Task 2.5：实现 check_endnote_export.py

**目标**：提供导入前健康检查，而不是把所有问题都压到 import 阶段。  
**重点**：格式合法性、缺字段、DOI 污染、未知 type、明显重复项。

#### Task 2.6：补 workflow 级测试

**目标**：锁定 EndNote import-first 端到端行为。  
**建议测试文件**：

- `tests/test_bib_render.py`
- `tests/test_endnote_import_workflow.py`

## Phase 3：兼容式扩展 `citation_mapping.py`

### 目标

仅在前两阶段稳定后，再做多来源 mapping 的兼容式扩展。

### 当前事实

当前 `core/citation_mapping.py` 明确是 Zotero 语义：

- `get_latex_key(zotero_key: str)`
- `get_or_create_latex_key(zotero_key: str)`
- 保存格式为 v1：`{"mappings": {"KeyA": "ref001"}}`

并且现有 `tests/test_zotero_extract.py` 已对这些行为建立回归约束。

### 要求

1. **旧格式必须可读**

```json
{
  "description": "Zotero citation-key to LaTeX ref number mapping",
  "version": "1.0",
  "mappings": {
    "old_zotero_key": "ref001"
  }
}
```

2. **旧 API 必须继续可用**

- `get_latex_key(zotero_key)`
- `get_or_create_latex_key(zotero_key)`

3. **新能力可以以兼容方式增加**

例如：

```python
get_or_create_latex_key(source_key: str, source_system: str = "generic") -> str
```

或提供新的通用方法，同时保留旧方法 wrapper。

4. **不强制立即升级所有旧 mapping 文件到新 schema**

新 schema 可以作为 v0.4 可选增强或内部支持能力，但不应成为主线稳定性的前置依赖。

### 退出标准

- 旧 Zotero mapping 可读
- 旧 Zotero 流程不报错
- EndNote 导入可复用或新增 `refNNN`
- 重复运行编号稳定

### 明确任务包

#### Task 3.1：只读方式确认旧行为约束

**目标**：以 `tests/test_zotero_extract.py` 和当前 `CitationMapping` 行为为准，列出不能破坏的 API。  
**重点**：

- `load()`
- `get_latex_key()`
- `get_or_create_latex_key()`
- `save()`
- `to_citation_lock_content()`

#### Task 3.2：设计兼容式扩展方案

**目标**：确定是“新增通用方法 + 旧方法保留”，还是“旧方法参数兼容扩展”。  
**要求**：不得破坏 Zotero 调用点。

#### Task 3.3：实现旧格式读取 + 新能力接入

**目标**：EndNote 导入可接入 mapping，但旧 JSON 结构仍可继续使用。  
**重点**：不强制 schema 迁移。

#### Task 3.4：补 mapping 稳定性测试

**目标**：覆盖重复运行、增量导入、旧格式兼容。  
**重点**：

- 同一输入两次运行不漂移
- 新记录从最大编号继续
- 旧记录不重排

## 4. 验收标准

## 4.1 功能验收

必须满足：

1. 能从 EndNote XML 导入中英文文献并生成 `refs-import.bib`
2. 能从 RIS 导入中英文文献并保留多作者
3. 相同 DOI 的重复项可自动去重
4. 未知 type 不会导致崩溃，而是 warning
5. `citation-mapping.json` 可重复运行且 `refNNN` 稳定
6. 不破坏现有 Zotero 命令与工作流
7. 有基本测试覆盖
8. 文档可直接给用户使用

## 4.2 关键回归验收

- `00-bib-zotero/check_bib_quality.py` 仍可运行
- `00-bib-zotero/sync_from_word.py` 仍可运行
- 旧版 mapping 文件仍可读取
- `CitationMapping` 旧 API 仍可用

## 4.3 稳定性验收

- 同一输入重复运行两次，mapping 不漂移
- 增量导入时旧编号不变，新编号从最大编号后继续
- `refs-import.bib` 不重复追加相同条目

## 5. 风险清单

### 风险 1：`citation_mapping.py` 改造破坏 Zotero

这是最高优先级风险。

**回避策略**：

- mapping 改造后置到 Phase 3
- 保留旧 API
- 回归测试先于重构存在

### 风险 2：EndNote 导出格式复杂度高于预期

**回避策略**：

- 先准备真实或半真实样例
- parser 对未知字段保留 raw，避免过早丢信息

### 风险 3：去重误合并

**回避策略**：

- 仅对高置信匹配自动合并
- 低置信匹配只 warning

### 风险 4：过早设计导致主线开发发散

**回避策略**：

- 先完成 MVP
- 把 v2 mapping schema 和更完整的多来源抽象视为后置增强
