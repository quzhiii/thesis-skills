# EndNote v0.4 Phase 0 预研备忘录

## 文档目的

本文档只做实现前预研，不涉及代码修改。目标是基于当前 `thesis-skills` 主线代码，明确：

- EndNote v0.4 可直接复用的 Zotero / core 主干
- 当前架构中不能被破坏的耦合点
- XML / RIS / BibTeX 样例数据应覆盖哪些场景
- 在真正实现前需要先锁定的风险

---

## 1. 观察到的可复用主干

### 1.1 项目发现与规则入口已经成型

从 `00-bib-zotero/sync_from_word.py`、`00-bib-zotero/check_bib_quality.py`、`core/project.py`、`core/rules.py` 可以确认，仓库已有稳定的“脚本 -> 规则包 -> 项目发现”主干：

- `find_rule_pack(repo_root, ruleset)` 负责定位规则包
- `ThesisProject.discover(...)` 负责发现主 tex、章节、bibliography_files、reports_dir
- `rules.yaml` 已允许 `ref/refs-import.bib` 作为 bibliography_files

这意味着 EndNote v0.4 没必要重新发明项目定位机制。最自然的策略是：

- 复用 `find_rule_pack()`
- 复用 `ThesisProject.discover()`
- 复用现有 reports 目录约定

### 1.2 报告输出主干已可复用

从 `core/reports.py` 可以确认，仓库已有统一 JSON 报告写出入口：

- `write_report(report_path, checker, ruleset, findings, extra_summary, extra_payload)`

它已提供：

- summary 统一结构
- findings 列表结构化输出
- PASS / FAIL 判定

这意味着 EndNote v0.4 的导入前检查器或导入后报告，不应另起一套完全不同的报告风格。至少 summary / findings 的组织方式应尽量向现有 `write_report()` 靠齐。

### 1.3 Bib 质量检查链路已存在

从 `00-bib-zotero/check_bib_quality.py` 与 `core/checkers.py` 可以确认：

- Bib 检查不是 Zotero 独占，而是面向 `.bib` 成品检查
- 当前检查逻辑包括：
  - unsupported entry type
  - missing langid
  - malformed doi

这意味着 EndNote v0.4 可以把输出目标聚焦到：

> 尽快产出合格的 `refs-import.bib`，然后继续复用现有 Bib 检查链路。

### 1.4 现有 examples 已体现 bib / report 组织方式

从 `examples/minimal-latex-project/`、`examples/minimal-latex-project-verify/`、`examples/migration-demo/` 可以观察到：

- 项目内已存在 `ref/refs.bib` 与 `ref/refs-import.bib` 双文件结构
- `reports/` 目录是标准输出位置
- 端到端 runner 测试依赖这些目录约定

这意味着 EndNote v0.4 的 import-first 输出路径无需创新，应直接沿用：

- `ref/refs-import.bib`
- `ref/citation-mapping.json`
- `reports/*.json`

## 2. 当前不能被破坏的耦合点

### 2.1 `CitationMapping` 目前明确是 Zotero 语义

从 `core/citation_mapping.py` 与 `tests/test_zotero_extract.py` 可以确认：

- `CitationMapping.load()` 默认读取 `ref/citation-mapping.json`
- `mappings` 当前结构为 `dict[str, str]`
- `get_latex_key(zotero_key)` / `get_or_create_latex_key(zotero_key)` 的参数语义明确是 Zotero key
- `save()` 当前写出的是 v1 风格 JSON：

```json
{
  "description": "Zotero citation-key to LaTeX ref number mapping",
  "version": "1.0",
  "mappings": {
    "KeyA": "ref001"
  }
}
```

这说明 EndNote v0.4 若要扩展 mapping，必须满足：

1. 旧 JSON 结构继续可读
2. 旧方法签名继续可用
3. 旧测试继续通过
4. 旧 `sync_from_word.py` 不需要同步重写

### 2.2 Zotero 同步脚本直接依赖 mapping.mappings 的原始结构

从 `00-bib-zotero/sync_from_word.py` 可见：

- 它直接把 `mapping.mappings` 传给 `compare_citations()`
- `compare_citations()` 的输入就是 `dict[str, str]`

这意味着：

> 一旦 `mapping.mappings` 从 `dict[str, str]` 变成嵌套对象结构，而调用点未同步兼容，Zotero workflow 会直接坏掉。

因此 EndNote v0.4 不应把“schema 升级”当成第一阶段前置条件。

### 2.3 `citation-lock.tex` 生成机制依赖当前 LaTeX key 结构

`CitationMapping.to_citation_lock_content()` 直接基于 `mappings.values()` 生成 `\nocite{ref001,...}`。

这意味着：

- `refNNN` 仍然是整个系统的稳定锚点
- EndNote 导入应复用现有 `refNNN` 体系，而不是另起 key 规则

### 2.4 当前测试体系是 `unittest` 风格

`tests/test_zotero_extract.py`、`tests/test_runner.py` 都是 `unittest` 风格。执行计划虽然用 `pytest` 作为运行方式没有问题，但新增 EndNote 测试最好与当前测试风格保持兼容。

## 3. 样例数据需求

## 3.1 XML 样例应覆盖的场景

最小 XML 样例建议包含：

1. 中文标题条目
2. 英文标题条目
3. 多作者条目（至少 3 位作者）
4. 有 DOI 条目
5. 无 DOI 条目
6. 期刊文章类型
7. 书籍或会议论文类型
8. 至少 1 条未知或非常见 type
9. 至少 1 条可能重复记录（同 DOI 或同 title/year/author）

原因：

- parser 需要尽早验证 title / authors / year / doi / type mapping
- canonicalize 需要验证 DOI、标题、作者处理
- match_refs 需要验证高置信与低置信重复规则

## 3.2 RIS 样例应覆盖的场景

RIS 样例除了覆盖与 XML 类似的内容，还必须额外覆盖：

1. `TY` / `ER` 边界正确性
2. 多个 `AU` / `A1` 作者
3. continuation line
4. 中英文混合条目
5. `DO`、`JO` / `JF`、`T2`、`PB`、`SN` 等常见 tag

原因：

- RIS 的格式风险主要不在字段含义，而在 tag 边界与文本续行

## 3.3 BibTeX fallback 样例应覆盖的场景

BibTeX 样例建议只作为 fallback / compatibility case，不必一开始追求复杂 parser：

1. 标准 `@article{...}` 条目
2. 含 DOI 但格式不够干净的条目
3. 缺 `langid` 的条目
4. 非常见 entry type 条目
5. 至少 1 条中文条目

原因：

- 当前仓库已有 Bib 质量检查链路
- v0.4 对 BibTeX 的目标是“接受并纳入 import-first”，不是重新发明完备 BibTeX parser

## 4. 当前真实风险

### 风险 1：把 EndNote 抽象设计放在 Zotero 兼容之前

若先追求完全 source-agnostic schema，而不是先保证旧 Zotero 行为不变，会优先破坏：

- `CitationMapping`
- `sync_from_word.py`
- `tests/test_zotero_extract.py`

### 风险 2：样例数据不够真实，导致 parser 设计脱离实际

如果 XML / RIS 样例过于理想化，后续实现阶段会出现：

- type mapping 假设不成立
- authors 结构与真实导出不符
- continuation line 处理不完整

### 风险 3：过早把 BibTeX 当成与 XML / RIS 同等复杂的 parser 任务

从当前代码基础看，BibTeX 更适合作为 fallback 输入，而不是 v0.4 最前期的重点 parser 任务。

### 风险 4：导入报告另起一套结构

`core/reports.py` 已经建立了统一报告写出风格。如果 EndNote 新增报告完全绕开现有 summary / findings 约定，后续维护会变差。

### 风险 5：在 examples/ 下过早添加复杂 EndNote 演示工程

当前 examples 已有最小 LaTeX 工程与 migration demo。实现前更合理的做法是先把 EndNote 样例放在 `tests/data/`，等 import-first 稳定后，再决定是否需要新增 example project。

---

## 5. 对 Phase 1 之前的建议

在真正开始编码前，建议先完成以下准备：

1. 固化 XML / RIS / BibTeX 三类最小样例标准
2. 写出 Zotero 不可破坏行为清单
3. 明确 EndNote 导入报告优先复用 `write_report()` 结构
4. 明确 BibTeX 在 v0.4 中是 fallback，而不是核心优先 parser
5. 进入实现阶段时优先使用独立 worktree，而不是直接在 `main` 工作区动实现
