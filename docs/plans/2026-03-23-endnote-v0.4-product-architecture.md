# EndNote v0.4 产品与架构说明

## 文档定位

本文档用于说明 `thesis-skills` 为什么要推进 EndNote v0.4、这一版的目标边界是什么、架构愿景是什么，以及 v0.4 / v0.5 的能力分层。

适用对象：

- 产品与版本规划负责人
- 需要理解改造方向的开发者
- 后续实施前做方案审阅的 AI agent / reviewer

---

## 1. 为什么做 EndNote v0.4

当前 `thesis-skills` 主线已经具备较完整的 Zotero 工作流：

- `00-bib-zotero/THESIS_BIB_ZOTERO.md`
- `00-bib-zotero/sync_from_word.py`
- `core/zotero_extract.py`
- `core/citation_mapping.py`

但 EndNote 目前仍处于弱支持状态。仓库内现有的 `00-bib-endnote/THESIS_BIB_ENDNOTE.md` 仅描述了一条非常轻量的兼容路径：

1. 从 EndNote 导出 BibTeX
2. 手工清理非标准类型
3. 写入 `ref/refs-import.bib`
4. 复用 `00-bib-zotero/check_bib_quality.py`

这意味着 EndNote 用户虽然能够“接入”，但还没有获得一等支持：

- 没有标准化导入入口
- 没有稳定的 XML / RIS 解析链路
- 没有统一导入报告
- 没有为后续增量同步预留清晰接口

因此，EndNote v0.4 的目标不是一次性追平 Zotero 全能力，而是先把 EndNote 用户稳定纳入当前主线的 check / fix 主干。

---

## 2. 当前仓库状态与问题判断

### 2.1 当前已存在的可复用主干

仓库已有以下基础能力可被 EndNote v0.4 复用：

- 参考文献 key 分配与持久化：`core/citation_mapping.py`
- Bib 质量检查：`00-bib-zotero/check_bib_quality.py`
- 项目与规则定位：`core/project.py`、`core/rules.py`
- 报告与检查主干：`core/checkers.py`、`core/reports.py`
- 规则包：`90-rules/packs/tsinghua-thesis/rules.yaml`
- Zotero 回归基线：`tests/test_zotero_extract.py`

### 2.2 当前主要问题

1. **EndNote 入口过弱**：只有说明文档，没有真正的 import CLI。
2. **内部抽象偏 Zotero 单源**：`CitationMapping` 仍以 `zotero_key -> refNNN` 为中心。
3. **缺少 EndNote 测试样本**：没有 XML / RIS 样例，也没有 EndNote 回归测试。
4. **缺少导入报告**：EndNote 用户无法知道缺字段、重复项、未知 type 等风险。

---

## 3. v0.4 的目标边界

### 3.1 v0.4 要完成的事情

EndNote v0.4 只解决 **import-first** 场景，即：

```text
EndNote XML / RIS / BibTeX
-> 解析
-> 归一化
-> 去重
-> 分配或复用 refNNN
-> 生成 refs-import.bib
-> 更新 citation-mapping.json
-> 输出 JSON report
```

本版目标包括：

- 支持从 EndNote XML / RIS / BibTeX 导入参考文献
- 自动归一化为内部统一结构
- 生成或更新 `ref/refs-import.bib`
- 生成或更新 `ref/citation-mapping.json`
- 输出导入与检查报告（JSON）
- 补齐样例、测试与文档
- 不破坏现有 Zotero 工作流

### 3.2 v0.4 明确不做的事情

本版不做：

- EndNote Word field-code parser
- EndNote Traveling Library 原生抽取
- EndNote `Word -> LaTeX` 增量同步
- GUI
- 重量级运行时依赖
- 为了 EndNote 大幅推倒现有 Zotero 主干

这条边界必须保持稳定。若后续实现中出现“需要先重写 Zotero 才能做 EndNote”的结论，应优先收缩方案，而不是扩大本版范围。

---

## 4. 核心产品策略：Import-First，而非 Sync-First

### 4.1 为什么采用 import-first

选择 import-first，有四个现实原因：

1. **更符合当前主线成熟度**：仓库已有 bib 检查与规则主干，但没有 EndNote Word 抽取基础。
2. **风险更低**：可以先在导入层验证 EndNote 数据质量、类型映射和去重策略。
3. **兼容主线节奏**：可以以较小改动接入现有 check / fix 体系。
4. **为 v0.5 预留接口**：一旦内部参考文献模型稳定，再补 EndNote Word 同步更稳。

### 4.2 本策略的成功标准

如果 v0.4 成功，EndNote 用户将能做到：

- 不再只能“手工导出 BibTeX + 手工清洗”
- 直接导入 XML / RIS / BibTeX
- 获得稳定的 `refNNN` 编号分配
- 在报告中看到重复、缺字段、未知类型等问题
- 继续复用仓库现有的 checker / fixer / rule-pack 主干

---

## 5. 架构愿景

### 5.1 中期愿景

`thesis-skills` 不应长期停留在 Zotero 单源模型，而应逐步演进为：

- Zotero
- EndNote
- 未来可能的 Mendeley / 通用 RIS / 其他来源

都可以接入的多来源参考文献工作流。

这要求内部抽象从“特定来源的 key”升级为“来源无关的 canonical reference”。

### 5.2 但 v0.4 的原则不是一次性做完全部抽象

尽管中期愿景是 source-agnostic，但 v0.4 必须遵循：

> 只在 EndNote import-first 需要的最小范围内引入通用抽象，不为了追求抽象完整性而提前重构主线。

尤其是 `core/citation_mapping.py`：

- 当前实现明确面向 Zotero
- 已被 `tests/test_zotero_extract.py` 和现有工作流使用
- 因此它是主线兼容性的最高风险点

结论是：

> v0.4 可以为多来源 mapping 演进预留接口，但不应以“彻底重写 mapping schema”为前提条件。

---

## 6. 建议的架构分层

### 6.1 导入层

负责读取 EndNote 导出结果：

- XML parser
- RIS parser
- BibTeX fallback parser / adapter

职责：

- 读取原始导出文件
- 抽取常见字段
- 保留原始记录片段
- 输出统一的中间对象

### 6.2 规范化层

负责把不同来源的记录规整为统一内部表示：

- DOI 归一化
- 标题清洗
- 作者名标准化
- `langid` 推断
- `canonical_id` 生成
- 类型映射

这一层是未来多来源扩展的基础。

### 6.3 匹配与编号层

负责：

- 去重
- 复用已有 mapping
- 分配新 `refNNN`

该层必须优先保证：

- 同一输入重复运行时稳定
- 增量导入时旧编号不漂移
- 旧 Zotero mapping 不被破坏

### 6.4 输出与报告层

负责：

- 生成 `ref/refs-import.bib`
- 输出 `reports/endnote_import-report.json`
- 记录 warnings / unsupported types / missing fields / dedupe 结果

## 7. v0.4 / v0.5 分层

### 7.1 v0.4

本版应交付：

- EndNote XML / RIS / BibTeX import
- 归一化
- 去重
- `refs-import.bib` 输出
- mapping 更新
- JSON report
- 文档、样例与测试

### 7.2 v0.5

后续再做：

- `core/endnote_extract.py`
- `00-bib-endnote/sync_from_word.py`
- EndNote Word field-code 读取
- Traveling Library 抽取
- 真正的 `Word -> LaTeX` 增量同步

这个分层的意义是：

- v0.4 先解决“把 EndNote 文献稳定带进系统”
- v0.5 再解决“如何从 Word 中识别哪些文献被引用、如何做增量同步”

---

## 8. 主线开发原则

在主线仓库推进 EndNote v0.4 时，应遵守以下原则：

1. **先稳住 Zotero，再扩 EndNote**
2. **先样例与测试，再谈大抽象**
3. **优先最小可运行链路，不做过度设计**
4. **所有新增规则都应服务于当前 import-first，而不是提前为 v0.5 铺满复杂逻辑**
5. **任何对 `citation_mapping.py` 的改动都必须有 Zotero 回归验证**

---

## 9. 成功定义

如果 EndNote v0.4 达标，应满足以下产品层结果：

- EndNote 用户可直接使用 XML / RIS / BibTeX 导入
- 输出 Bib 与 mapping 可重复运行且稳定
- 错误与警告可见、可读、可定位
- 现有 Zotero 工作流不被破坏
- 仓库内部开始形成可扩展的多来源参考文献基础
