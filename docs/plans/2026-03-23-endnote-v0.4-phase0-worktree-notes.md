# EndNote v0.4 Phase 0 Worktree Notes

## Task 0.2：Zotero 复用点与不可破坏耦合

以下结论全部基于当前 worktree 中的只读代码检查得出。

### A. 可直接复用的主干

#### 1. 项目发现主干可直接复用

`core/project.py` 中的 `ThesisProject.discover()` 已经负责：

- 发现主 tex 文件
- 收集章节文件
- 收集 bibliography_files
- 创建 `reports/` 目录

这意味着 EndNote v0.4 不需要再设计一套新的项目定位逻辑。`import_library.py` 和 `check_endnote_export.py` 应优先复用同一发现流程。

#### 2. 规则包入口可直接复用

`00-bib-zotero/sync_from_word.py` 当前通过 `find_rule_pack(repo_root, args.ruleset)` 加载规则包，并把 pack 中的 `main_tex_candidates`、`chapter_globs`、`bibliography_files` 传给 `ThesisProject.discover()`。

这说明 EndNote v0.4 最自然的路径是：

- 继续用 `find_rule_pack()`
- 继续用 `pack.rules["project"]` 中的项目发现参数
- 不引入新的规则入口协议

#### 3. 报告输出模式可直接复用

`core/reports.py` 的 `write_report()` 已建立统一 JSON 输出格式：

- `summary`
- `findings`
- `extra_summary`
- `extra_payload`

因此 EndNote v0.4 的健康检查报告与导入报告，不应完全另起一套风格。即使导入报告需要额外字段，也应尽量保持 summary / findings 的现有组织方式。

#### 4. Bib 成品检查链路可直接复用

`00-bib-zotero/check_bib_quality.py` 只是一个很薄的入口，真正逻辑在 `core/checkers.py::run_bib_quality_check()`。

该检查器当前已经覆盖：

- `unsupported_entry_type`
- `missing_langid`
- `malformed_doi`

因此 EndNote v0.4 的短期目标应是：

> 尽快生成足够稳定的 `ref/refs-import.bib`，然后继续使用现有 Bib 质量检查链路。

---

### B. 当前不能破坏的耦合点

#### 1. `CitationMapping` 仍是纯 Zotero 语义

`core/citation_mapping.py` 目前明确以以下假设工作：

- `mappings: dict[str, str]`
- key 是 Zotero key
- value 是 `refNNN`
- `get_latex_key(zotero_key)`
- `get_or_create_latex_key(zotero_key)`

这不是“隐含如此”，而是显式 API 设计如此。

#### 2. Zotero 同步脚本直接消费 `mapping.mappings`

`00-bib-zotero/sync_from_word.py` 中：

- `comparison = compare_citations(extraction, mapping.mappings)`

而 `compare_citations()` 当前期望的也是 `dict[str, str]`。

所以如果 EndNote 直接把 `mapping.mappings` 变成嵌套结构或复合对象，Zotero 工作流会立即失效。

#### 3. `citation-lock.tex` 机制依赖现有 key 体系

`CitationMapping.to_citation_lock_content()` 直接把 `mappings.values()` 作为 `\nocite{ref001,...}` 的来源。

这说明：

- `refNNN` 是当前稳定锚点
- EndNote v0.4 应复用这个锚点，而不是发明新的 latex key 体系

#### 4. 现有测试已经锁定 Zotero 行为

`tests/test_zotero_extract.py` 已对这些行为建了回归约束：

- `CitationMapping.load()`
- `CitationMapping.get_or_create_latex_key()`
- `CitationMapping.save()`
- `CitationMapping.to_citation_lock_content()`
- `compare_citations()` 的 key 集合比较逻辑

这意味着 EndNote 实作阶段一旦动到 `citation_mapping.py`，必须以这些测试为硬边界。

## Task 0.3：examples 约束与输入标准

### A. examples 对 EndNote 的启示

#### 1. 当前项目结构已经默认支持 `ref/refs-import.bib`

`90-rules/packs/tsinghua-thesis/rules.yaml` 中，`project.bibliography_files` 已包含：

- `ref/refs.bib`
- `ref/refs-import.bib`

说明 EndNote v0.4 无需争取新的 bibliography file 位置。

#### 2. 当前 example 里已有 intentionally imperfect 的 `refs-import.bib`

`examples/minimal-latex-project/ref/refs-import.bib` 只有一条：

```bibtex
@article{importedmissinglang,
  title = {Imported Missing Langid},
  author = {Smith, Alex},
  year = {2022}
}
```

这说明 examples 当前已经在用 `refs-import.bib` 承载“导入来源文献”，且允许其先带问题再进入 checker。

对 EndNote v0.4 的启示是：

- import-first 的第一目标是“进入系统并可被检查”
- 不要求导入阶段一次性完美清洗所有字段

#### 3. runner 测试依赖 examples 的稳定结构

`tests/test_runner.py` 直接使用：

- `examples/minimal-latex-project/`

并依赖 `reports/run-summary.json` 的输出。因此在 v0.4 初期，最好不要贸然扩大 examples 的复杂度。更合理的方式是：

- parser 样例优先放到 `tests/data/`
- example project 只在需要做端到端演示时再补

### B. Task 0.4 样例数据标准

#### XML 最小样例标准

必须覆盖：

1. 中文标题
2. 英文标题
3. 多作者
4. 有 DOI
5. 无 DOI
6. 期刊与非期刊至少各一类
7. 至少一条未知或非常见 type
8. 至少一组高置信重复（同 DOI）

#### RIS 最小样例标准

必须覆盖：

1. `TY` / `ER` 边界
2. 多作者 `AU` / `A1`
3. continuation line
4. 中英文混合内容
5. `DO` / `JO` / `JF` / `T2` / `PB` / `SN` / `LA`
6. 至少一条 type mapping 模糊项

#### BibTeX fallback 最小样例标准

必须覆盖：

1. 标准 article 条目
2. 带 DOI 但格式不干净的条目
3. 缺 `langid` 的条目
4. 至少一条中文条目
5. 至少一条非常见 entry type

并明确：

> v0.4 中 BibTeX 是兼容输入，不是第一优先级核心 parser 战场。

---

## 当前收口结论

1. Task 0.2 已足够明确：EndNote v0.4 应复用项目发现、规则包、报告输出、Bib 检查主干。
2. 当前最高风险点仍是 `core/citation_mapping.py` 与 `sync_from_word.py` 之间对 `dict[str, str]` 的直接耦合。
3. Task 0.3 / 0.4 的最佳执行方式，是先在 `tests/data/` 准备 XML / RIS / BibTeX 样例，而不是先改 examples/。
