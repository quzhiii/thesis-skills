# Thesis Skills v1.1 Citation Integrity Upgrade Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship a local-first, deterministic Citation Integrity Engine that upgrades `10-check-references` from basic cite-key checking into a report-first pre-submission citation safety layer.

**Architecture:** Keep the public entrypoint at `10-check-references/check_references.py`, but move non-trivial parsing and lint logic into `core/citation_integrity/` so v1.2–v2 can build on stable internal contracts. Preserve `reports/check_references-report.json` for pipeline compatibility and additionally emit `reports/citation-integrity-report.json` as the v1.1 product artifact. Readiness should continue to consume normal checker reports, with a narrow adapter path for the richer Citation Integrity status.

**Tech Stack:** Python standard library, existing `core.common.Finding`, existing `core.reports.write_report`, existing `ThesisProject` discovery, pytest, local LaTeX/BibTeX fixtures only. No LLM, no external API, no automatic citation insertion or rewriting.

## 1. Source Context Read

This plan consolidates:

- `docs/plans/thesis-skills-citation-integrity-roadmap.md`
- `docs/plans/agent-prompt-thesis-skills-v1.1-citation-integrity.md`
- Current implementation in `10-check-references/check_references.py`, `core/checkers.py`, `core/readiness_gate.py`, `core/compile_parser.py`, `run_check_once.py`, and `docs/modules.md`

Current reality:

- `10-check-references/check_references.py` is a thin CLI wrapper around `core.checkers.run_reference_check`.
- `run_reference_check` currently handles cite-key existence, unused bib entries, and duplicate titles with simple regexes.
- `core.checkers.run_format_check` already owns label/ref checks.
- `core.compile_parser` already identifies missing citations from compile logs, but through the compile report path.
- `core.readiness_gate` grades the References dimension from `check_references` errors/warnings.
- There are no dedicated citation integrity tests or demo projects yet.

Key planning implication: v1.1 should avoid duplicating the format and compile layers blindly. It should consolidate citation-facing evidence while preserving existing reports and pipeline compatibility.

## 2. Brainstormed Approaches

### Option A — Minimal Patch Inside `core/checkers.py`

Add duplicate key, field lint, DOI/year lint, log parsing, and crossref checks directly inside `run_reference_check`.

**Pros:** Fastest; fewest files; low plumbing overhead.
**Cons:** `core/checkers.py` is already broad; v1.2/V2 will become painful; parser behavior hard to test independently; higher risk of regressions.

### Option B — New `core/citation_integrity/` Engine, Existing CLI Wrapper

Create dedicated modules for TeX parsing, BibTeX parsing, field linting, log evidence, crossref lint, report shaping, and readiness adaptation. `run_reference_check` delegates to the engine.

**Pros:** Best long-term shape; testable units; aligns with V1.1–V5 roadmap; keeps CLI stable; supports richer product artifact.
**Cons:** More files; requires careful compatibility bridge to existing `write_report` and readiness gate.

### Option C — Separate `18-citation-integrity/` Module

Create a new top-level module and keep `10-check-references` mostly as-is.

**Pros:** Product positioning is explicit.
**Cons:** Splits reference checking into two user-facing paths; breaks roadmap instruction to build on `10-check-references`; more documentation and runner complexity.

### Decision

Choose **Option B**.

It is the best balance: v1.1 remains compatible with the current CLI and readiness gate, while creating enough structure for V1.2 reports, V2 external metadata verification, and V3 hallucination-risk screening.

## 3. Scope Boundaries

### v1.1 Must Do

- Local `.tex` citation parsing across common citation commands.
- `.bib` entry parsing with duplicate key detection across one or more bib files.
- Missing cited key detection.
- Unused bibliography detection.
- Required field linting: base fields plus type-specific fields.
- DOI shape linting without network access.
- Year shape linting with configurable current-year tolerance.
- Optional compile-log citation-warning ingestion, reusing current parser behavior where practical.
- Label/ref cross-reference evidence without letting it dominate citation status.
- Stable JSON artifact: `reports/citation-integrity-report.json`.
- Compatibility report: `reports/check_references-report.json` must still work for `run_check_once.py` and readiness.
- Broken demo project and tests.

### v1.1 Must Not Do

- CrossRef, OpenAlex, Semantic Scholar, PubMed, DBLP, or Zotero Web API calls.
- LLM usage.
- AI fake-reference judgment.
- Claim-citation support analysis.
- Citation recommendation.
- Automatic replacement or insertion of citations/references.
- Broad Word field-code parsing.
- Large unrelated refactors.

### Phase 2 — Local Integrity Lints

Objective: implement the v1.1 local safety checks.

Files:

- Create: `core/citation_integrity/field_lint.py`
- Create: `core/citation_integrity/cross_reference_lint.py`
- Create: `core/citation_integrity/latex_log_parser.py` or thin adapter over `core.compile_parser`
- Test: `tests/test_citation_integrity_bib_fields.py`
- Test: `tests/test_citation_integrity_latex_log.py`

Tasks:

1. Missing cited key => `BLOCK`.
2. Unused bibliography entry => `WARN` or `INFO` depending default policy; use `WARN` for v1.1 service value.
3. Duplicate key:
   - identical metadata => `WARN`
   - same key, materially different metadata => `BLOCK`
4. Required fields:
   - all entries: `title`, `author`, `year`
   - `article`: `journal` or `journaltitle`
   - `inproceedings`: `booktitle`
   - `book`: `publisher`
5. DOI:
   - missing DOI => `WARN`, never `BLOCK`
   - malformed DOI => `WARN`
   - Chinese/non-DOI references should not be labeled fake or blocking.
6. Year:
   - missing or non-numeric => `WARN`
   - future beyond current year + 1 => `WARN`
7. Compile log:
   - undefined citation => `BLOCK` in citation integrity if present in log evidence.
   - empty bibliography/backend warnings => `WARN` unless clearly undefined citation.
8. Crossrefs:
   - missing label target => `WARN` or `BLOCK` only if the roadmap decides references dimension should block; recommended: `WARN` in citation integrity, keep hard blocking in format checker.
   - unused labels => `INFO` or low `WARN`.

Completion signal: all lint categories have focused unit tests and stable issue IDs.

## 6. V1.2–V5 Roadmap After v1.1

### v1.2 — Report UX and Service Packaging

Ship after v1.1 is stable.

- Markdown report.
- CSV risk list.
- Clean vs broken examples.
- README/showcase preview.
- Service-facing sample report screenshots.
- 3–5 real thesis/project samples tested privately.

Decision gate: do users understand and pay for “引用安检报告” before external verification exists?

### v2.0 — External Metadata Verification

Only after v1.1 local evidence is trusted.

- CrossRef DOI/title lookup.
- OpenAlex title lookup.
- Optional Semantic Scholar/PubMed by domain.
- Local cache, rate-limit handling, offline fallback.
- Evidence objects with source, timestamp, candidates, match score.

Decision gate: English DOI/title matching is stable and network failure never breaks local checks.

### v3.0 — Hallucination-Risk Screening

Built on v2 evidence, not on LLM claims.

- Risk states: `PASS`, `WARN`, `REVIEW`, `HIGH_RISK`, `UNSUPPORTED`.
- High-risk CSV.
- Conservative language: never say “this is fake”; say evidence was not found in enabled sources.
- Chinese/grey literature defaults to `UNSUPPORTED`, not `HIGH_RISK`.

Decision gate: false-positive handling is good enough for paid service delivery.

### v4.0 — Claim-Citation Support Triage

This is the first LLM-assisted stage, but still evidence-first.

- Extract local claim/citation clusters.
- Link to title/abstract/keywords where available.
- Triage mismatch/overclaim/weak support/citation-needed.
- No auto rewrite.
- Every output says “requires human review”.

Decision gate: output helps reviewers prioritize work without pretending to be final judgment.

### v5.0 — Candidate Citation Recommendation

Recommendation is human-confirmed and evidence-backed.

- Recommend from local library first.
- Expand from seed papers second.
- External query search third.
- Never use LLM-generated reference as a source.
- Export candidate references separately; never auto-insert.

Decision gate: candidate quality is high enough that users repeatedly accept/review suggestions.

## 8. Main Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Scope creep into V2–V5 | Enforce local-only/no-API/no-LLM gate in v1.1 tests and docs |
| Report schema conflicts with existing `write_report` | Emit both compatibility checker report and rich product report |
| Duplicate logic with format/compile checks | Reuse or adapt; do not silently create contradictory severities |
| BibTeX regex parser edge cases | Keep parser conservative, document supported syntax, add fixtures before widening |
| Chinese/non-DOI references misclassified | DOI missing is WARN only; existence verification deferred to v2/v3 with UNSUPPORTED state |
| Readiness regression | Preserve old fallback behavior and test both old/new report paths |
