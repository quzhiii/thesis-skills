## 2. 痛点区块 (Pain Points)

### 区块标题
写论文时，这些场景是否熟悉？

### 区块描述
Thesis Skills 不是 AI 写作工具，而是一套确定性工作流——解决论文写作中最实际、最耗时的环节。

### 四个痛点

**痛点 1：导师只审 Word，但论文是 LaTeX**
LaTeX 排版的论文无法直接给导师批注，手动转换耗时且容易丢失格式。需要一个审阅友好的导出副本。

**痛点 2：LaTeX 编译报错看不懂**
原始编译日志冗长且晦涩，关键错误 buried 在数百行输出中。难以快速定位真正需要修复的问题。

**痛点 3：修改意见分散，难以追踪**
导师的批注、邮件反馈、聊天记录散落在各处。新一轮修改时，不知道哪些已处理、哪些仍待办。

**痛点 4：提交前心里没底**
盲审或正式提交前，不确定引用是否完整、格式是否合规、语言是否通顺。缺少一个清晰的 go / no-go 判断。

## 3. 亮点区块 (Highlights)

### 区块标题
不只是解决问题，还能显著提升效率

### 区块描述
Thesis Skills 将论文写作中最耗时、最容易出错的环节自动化，同时保持确定性和可检查性。

### 亮点 1：显著的工作流提速
基于实际用户数据的效率对比，最耗时的环节实现了数量级的提升。
- **42×** 日常引用更新提速（85 分钟 → 2 分钟）
- **5.25×** Word→LaTeX 迁移提速（210 分钟 → 40 分钟）
- **1.8×** 新高校格式适配提速（100 分钟 → 56 分钟）

### 亮点 2：多学校模板一键适配
内置清华大学研究生毕业论文模板，支持通过 YAML 规则包自定义其他高校格式要求。
- 清华大学研究生模板（已内置）
- 通用高校论文模板
- 期刊投稿模板
- 自定义规则包支持

### 亮点 3：LaTeX 格式自动检查与修复
根据学校模板和写作指南，自动检查格式问题并生成安全修复建议，避免手动逐个调整。
- 图表、标题、居中格式检查
- 必需章节完整性验证
- 安全修复 + dry-run 预览
- 规则驱动，可自定义

### 亮点 4：引用管理自动化
从 EndNote / Zotero 自动导入文献，保持引用编号稳定，避免手动映射错误。
- EndNote XML/RIS/BibTeX 导入
- Zotero 与 Word 同步
- Citation Lock 编号锁定
- 去重与规范化

## 4. 场景卡片区块 (Scenario Cards)

### 区块标题
四个核心工作流

### 区块描述
每个场景都对应论文写作中的真实环节，产出明确、可检查的 artifact。

### 场景一：Word 草稿 → 结构化 LaTeX 工作流
从 Word 文档中提取引用结构，迁移到 LaTeX 项目中。保留引用编号稳定性，生成可审计的映射报告。

产物：📦 citation-map.json + refNNN 锁定

### 场景二：LaTeX 论文 → 导师审阅版 Word
将 LaTeX 论文导出为审阅友好的 .docx 格式。显式报告不支持的结构，不隐藏降级风险。

产物：📄 thesis-review.docx + export-limitations.json

### 场景三：原始编译日志 → 可读诊断报告
解析现有 .log 文件，将原始 TeX 报错转换为结构化 findings，区分阻塞性错误和警告。

产物：📊 compile-findings.json + run-summary.json

### 场景四：最终论文包 → 预提交裁决
汇总已有检查、修复、编译、导出、审阅产物，输出明确的 PASS / WARN / BLOCK 判断，并把未清审阅债务讲清楚。

产物：✅ readiness-gate.json (PASS | WARN | BLOCK)

## 5. 产物展示区块 (Artifact Showcase)

### 区块标题
每一步都有明确的产物

### 区块描述
Thesis Skills 的核心设计原则是"产物可检查"——每个工作流步骤都生成结构化报告，而非静默修改。

### 产物 1：review-friendly.docx
审阅友好的 Word 导出，附带显式限制报告。 unsupported constructs 被明确列出，而非静默降级。

预览内容：
```
# 导出摘要
✓ 正文段落: 142 段
✓ 数学公式: 89 个 (保留为 OMML)
△ 复杂表格: 3 个 (简化排版)
✗ TikZ 图形: 2 个 (需手动重绘)
---
显式报告限制，不隐藏降级
```

### 产物 2：compile-findings.json
将原始 LaTeX 编译日志转换为结构化 findings，保留原始日志片段以便溯源，同时提供可读的错误解释。

预览内容：
```json
{
  "severity": "error",
  "source": "thesis.tex:214",
  "message": "Undefined control sequence",
  "hint": "检查 \\newcommand 定义",
  "raw_log_excerpt": "..."
}
```

### 产物 3：review-todo.json
审阅闭环产物：将分散的反馈归一化为结构化 TODO，支持追踪 revision round 的处理状态，并为 chapter-level 摘要和后续 gate 解释提供边界清晰的输入。

预览内容：
```json
[
  {
    "id": "REV-003",
    "status": "pending",
    "source": "导师批注 P.12",
    "category": "论述深度",
    "action": "补充实验对比"
  }
]
```

### 产物 4：readiness-gate.json
预提交 Gate：汇总所有已有产物，给出明确的 PASS / WARN / BLOCK 裁决，以及具体的下一步行动，并把 review debt 的来源解释得更清楚。

预览内容：
```json
{
  "verdict": "WARN",
  "mode": "advisor-handoff",
  "blockers": [],
  "warnings": [
    "3 个深度语言问题需审阅"
  ],
  "next_action": "优先处理 warnings"
}
```

## 6. 可信区块 (Trustworthy)

### 区块标题
有边界、可检查、不夸大

### 区块描述
Thesis Skills 不是 AI 论文写作助手，也不是一键完成工具。它的价值在于确定性、可重复性和诚实性。

### 六点可信理由

1. **有边界的工作流**
每个功能都有明确的输入、输出和限制。不支持的构造会被显式报告，不会静默降级或假装处理。

2. **显式报告**
检查器不直接修改源文件，而是生成结构化 JSON 报告。修复器读取报告后生成补丁，支持 dry-run 预览。

3. **可检查的产物**
每个工作流步骤的产物都是人类可读的 artifact。你可以打开 JSON 文件查看具体发现，而不是依赖隐藏的内部状态。

4. **没有"魔法"**
不声称能"自动理解导师意见"或"智能修复所有问题"。含糊或高判断成本的修改保持 review-gated，由人做最终决定。

5. **局限性被暴露而非隐藏**
导出限制、编译解析边界、语言检查的覆盖范围——所有这些都被明确文档化，并在产物中显式报告。

6. **开源可审计**
MIT 协议开源，所有规则、检查逻辑和修复策略都可在代码和 YAML 规则包中查看和自定义。

## 设计原则

1. **用户语言优先**：避免使用 "checkers", "fixers", "YAML rule packs", "orchestration" 等内部术语
2. **诚实不夸大**：明确展示 v0.7 的边界，不承诺尚未实现的功能
3. **产物导向**：强调每个工作流的 artifact 产出，而非抽象功能描述
4. **学术可信**：设计风格简洁、专业，避免 flashy startup 风格
5. **移动友好**：响应式布局，确保在手机上的可读性
